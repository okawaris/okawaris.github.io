acquire（获取独占锁）

  • tryAcquire():尝试去获取独占锁，AQS在这里没有对其进行功能的实现，只有一个抛出异常的语句，用户可以对其重写实现公平锁、不公平锁、可重入锁、不可重入锁。

  • addWaiter(Node.EXCLUSIVE):一旦尝试获取锁未成功，就要使用该方法将一个waiting in exclusive mode的节点node其加入同步队列尾部。

  • acquireQueued():一旦加入同步队列，就需要使用该方法，自旋阻塞唤醒来不断的尝试获取锁，直到被中断或获取到锁。

    如上，AQS并未对tryAcquire方法做出实质性的实现，如果子类没有实现，则抛出UnsupportedOperationException，以下列举ReentrantLock中的公平锁和非公平锁对tryAcquire的实现：

    如上所示：FairSync#tryAcquire实现了公平锁（hasQueuedPredecessors方法）和可重入锁

  • java.util.concurrent.locks.ReentrantLock.NonfairSync#tryAcquire

release(释放独占锁)

例：ReentrantLock

java.util.concurrent.locks.ReentrantLock.Sync#tryRelease

如上，ReentrantLock.Sync#tryRelease方法处理步骤：

  • 判断当前线程是否是独占锁的拥有线程
  • 队列同步器AQS的state做减法操作
  • 队列同步器的state等于0时，标记当前独占锁的拥有线程为null
  • 返回true

acquireShared(获取共享锁)

  • tryAcquireShared，尝试添加共享锁。
  • 如果tryAcquireShared返回值<0,则执行doAcquireShared()
    • addWaiter(Node.SHARED)，将一个waiting in shared mode的节点node添加到同步队列的尾部.
    • 自旋

其中tryAcquireShared方法如果需要同步锁需要支持共享锁，则需要实现，如果不需要则无需实现，

例：CountDownLatch

看其await()方法：

CountDownLatch#await调用的是acquireSharedInterruptibly(1)(获取共享锁-线程中断优先版)

再看其countDown()方法：

CountDownLatch#countDown方法刚好调的就是releaseShared()(释放共享锁)，而且继续看CountDownLatch中的tryReleaseShared方法所做的操作是：将队列同步器中的state做-1操作。

所以CountDownLatch是共享锁

releaseShared(释放共享锁)

看了源码发现，releaseShared(释放共享锁)执行unparkSuccessor()的时机和release(释放独占锁)执行的时机是不一样的，releaseShared执行时，保证当前队列同步器中的节点只有一个（通常情况下是主线程），此时获得节点的下一个节点为空，自然就执行本节点(通常情况下是主线程)里的thead了；release(释放独占锁)执行时，不关心队列同步器（AQS）中的节点个数，所以会唤醒h的下一个节点里的thead（如果有的话），如果只有一个节点，则唤醒本节点。

下面看一段CountDownLatch的应用场景

如上，申请了个数为10的CountDownLatch，子线程有9个，主线程有1个，9+1刚好等于10.主线程迅速的启动了9个线程，然后自己开始await()(CountDownLatch#await方法其实是一个加共享锁的过程)，9个子线程开始countDown，当第9个子线程执行完毕countDown后，会触发unparkSuccessor()操作，此时队列同步器AQS中只剩下一个节点(主线程)，自然就只能唤醒主线程了。

java.util.concurrent.locks.AbstractQueuedSynchronizer#state的值代表队列同步锁的状态，不同的同步锁中所代表的含义不同

  • CountDownLatch中代表个数

  • 在ReentrantLock中代表当前施加在同步锁上的次数

    state初始化为0，表示未锁定状态。A线程lock时，会调用tryAcquire独占该锁并将state+1，此后，其他线程再tryAcquire时就会失败，直到A线程unlock到state=0（即释放锁）为止，其它线程才有机会获取该锁。当然，释放锁之前，A线程自己是可以重复获取此锁的（state会累加），这就是可重入的概念。但要注意，获取多少次就要释放多么次，这样才能保证state是能回到零态的。
