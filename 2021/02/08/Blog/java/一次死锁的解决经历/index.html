一次死锁的解决经历
2021-02-08|javamysql

最近业务系统出现了一些报警，查看日志如下

分析上面日志得出，产生了死锁。

联系DBA导出数据库死锁日志（通过SHOW ENGINE INNODB STATUS可实现），如下：

重点分析以下两段日志：

事务1

分析下事务一的日志的关键信息如下：

TRANSACTION 2310540：事务编号为2310540；
2 row lock(s)：该事物目前持有2条记录的锁；
更新的记录是：id = 586768
想要得到获得X锁

事务2:

分析下事务二的日志的关键信息如下：

TRANSACTION 2310539：事务编号为2310539；
49882 row lock(s)：该事物目前持有49882条记录的锁；
更新的记录是：id = 586774
持有X锁（该锁就是事务一想要得到的锁）
期望获得X锁

这里不得不提两个点：

  • 更新唯一索引会先获取S锁（待确认项）
  • 一般查询不会获得S锁（一般查询即：SELECT * FROM tablename where id = 53），之前我以为只要是spring有事务就会获得S锁，其实是错误的，如果不显式的使用共享锁，一般的查询是不会获取S锁的。

综上，我们可以描述时间轴为以下：

如上图所示，解决办法有两个：

  • 方法一：在事务二worker调度处，设置只更新合同状态，不要连带更新合同的所有扩展属性（可以在此将不需要更新的属性置空）
  • 方法二：将更新合同方法加上全局唯一锁，可以很大程度上避免死锁冲突，但是依然不可以完全避免，因为介于spring事务的传播特性，如果在更新合同扩展属性的外层调用（更新合同方法）有事务的话，那么事务的提交会交给上层方法调用结束后提交，因此，为了根本上解决这个死锁问题，还是得方法一（取决于如何平衡简单和严谨了）
